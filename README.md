# رُواء (Rawaa) — مكتبة رقمية للأطفال

الواجهة: **React + Vite + Tailwind CSS**
الباك-إند: **Supabase** (قاعدة بيانات PostgreSQL + مصادقة Auth جاهزة، بدون سيرفر تديره بنفسك).

## 1) إنشاء مشروع Supabase

1. روح إلى [supabase.com](https://supabase.com) → أنشئ حساب ومشروع جديد (اختر منطقة قريبة، مثل Frankfurt أو Singapore).
2. من **Project Settings → API** انسخ:
   - `Project URL`
   - `anon public` key
   - `service_role` key (سرّي — لا تشاركه ولا تحطه بالواجهة الأمامية أبداً)

## 2) تجهيز قاعدة البيانات

1. من لوحة Supabase: **SQL Editor → New query**.
2. الصق محتوى ملف **`supabase/schema.sql`** كامل، واضغط **Run**.
   هذا ينشئ الجداول التالية مع صلاحيات أمان (RLS) تضمن أن كل ولي أمر يشوف بيانات أطفاله فقط:
   - `profiles` (بيانات ولي الأمر — تُنشأ تلقائياً عند التسجيل)
   - `children` (ملفات الأطفال)
   - `stories` + `story_pages` (مكتبة القصص)
   - `favorites` (المفضلة لكل طفل)
   - `reading_progress` (تقدم القراءة لكل طفل)

## 3) (اختياري الآن) تفعيل رمز التحقق (OTP) بالبريد

> **ملاحظة:** لأغراض التجربة الحالية، تقدر تتخطى هذي الخطوة بالكامل — شوف قسم **"تخطي التحقق بالبريد للتجربة"** تحت. ارجع لهذا القسم وقت ما تجهز فعلياً لإرسال بريد حقيقي للمستخدمين.

بشكل افتراضي يرسل Supabase **رابط تأكيد** بالبريد، لكن واجهتنا مبنية على **رمز من 4 أرقام** (نفس تصميم الشاشات الأصلية). لتفعيل ذلك:

1. **Authentication → Emails → SMTP Settings** — فعّل Custom SMTP (Supabase الآن يمنع تعديل القوالب بدون SMTP). أسهل مزود مجاني: [Resend](https://resend.com) — أنشئ حساب، سوّي API Key، واستخدم:
   - Host: `smtp.resend.com` — Port: `465` — Username: `resend` — Password: مفتاح Resend (`re_...`) — Sender: `onboarding@resend.dev`
   - ⚠️ بدون توثيق نطاق بـ Resend، الإيميلات ترسل فقط لنفس إيميل حساب Resend — كافي للتجربة، وتحتاج توثيق نطاق لاحقاً وقت الإنتاج.
2. **Authentication → Emails → Confirm signup** template — عدّل القالب ليستخدم `{{ .Token }}` بدل رابط `{{ .ConfirmationURL }}`.
3. تأكد أن **Authentication → Providers → Email → Confirm email** مفعّل.

## تخطي التحقق بالبريد للتجربة

عشان تختبر الموقع فوراً بدون إعداد بريد إلكتروني:

1. **Authentication → Providers → Email** → عطّل خيار **Confirm email**.
2. الآن أي تسجيل حساب جديد من الموقع يصير مؤكَّد تلقائياً، ويتخطى شاشة "تحقق من بريدك" تلقائياً.

(بالكود: صفحة التسجيل تحوّل المستخدم للصفحة الرئيسية دائماً، والنظام يقرر الوجهة الصحيحة تلقائياً حسب حالة الحساب — فتشتغل بشكل سليم سواء كان التحقق مفعّل أو لا، بدون أي تعديل إضافي بالكود.)

## 4) صلاحية الأدمن + حسابات تجريبية جاهزة

1. شغّل **`supabase/002_admin.sql`** بنفس طريقة `schema.sql` (SQL Editor → لصق → Run). يضيف عمود `is_admin` وصلاحيات تسمح للأدمن يشوف كل الحسابات.
2. عبّي بملف `.env` (اختياري، له قيم افتراضية):
   ```
   ADMIN_EMAIL=admin@rawaa.app
   ADMIN_PASSWORD=Admin123456!
   TEST_GUARDIAN_EMAIL=guardian@rawaa.app
   TEST_GUARDIAN_PASSWORD=Guardian123456!
   ```
3. شغّل:
   ```bash
   npm run seed            # لو ما سويتها قبل — يعبي كل القصص
   npm run seed:accounts   # ينشئ حساب أدمن + حساب ولي أمر فيه 3 أطفال ببيانات متنوعة
   ```

هذا السكربت يستخدم Admin API فينشئ الحسابات **مؤكَّدة تلقائياً** (يتخطى خطوة البريد بالكامل بغض النظر عن إعدادات SMTP)، ويضيف لولي الأمر التجريبي 3 أطفال بأعمار ومستويات قراءة واهتمامات مختلفة، مع مفضلة وتقدم قراءة تجريبي، عشان تشوف كل الشاشات (لوحة التحكم، التقارير...) وفيها بيانات فعلية.

**حساب الأدمن** يوصل لصفحة **لوحة تحكم الأدمن** (رابط "لوحة الأدمن" يظهر بشريط التنقل تلقائياً بعد تسجيل الدخول بحساب أدمن) — تعرض عدد أولياء الأمور، الأطفال، القصص، وقائمة كل الحسابات المسجّلة.

## 4) متغيرات البيئة

```bash
cp .env.example .env
```

عبّي داخل `.env`:

```
VITE_SUPABASE_URL=...          ← Project URL
VITE_SUPABASE_ANON_KEY=...     ← anon public key
SUPABASE_URL=...                ← نفس الـ Project URL (لسكربت التعبئة فقط)
SUPABASE_SERVICE_ROLE_KEY=...   ← service_role key (لسكربت التعبئة فقط)
```

## 5) تثبيت الحزم وتعبئة كل القصص

```bash
npm install
npm run seed     # يرفع كل القصص (٥٢ حالياً) من src/data/stories.json إلى Supabase
```

## 6) التشغيل محلياً

```bash
npm run dev
```

## 7) البناء والنشر

```bash
npm run build
```

ارفع مجلد `dist/` (أو اربط الريبو) لـ Vercel/Netlify. لا تنسَ تضيف نفس متغيرات `VITE_SUPABASE_URL` و `VITE_SUPABASE_ANON_KEY` بإعدادات المشروع على المنصة (بدون متغيرات الـ `service_role`، هذي فقط للتعبئة المحلية).

## هيكلة المشروع

```
supabase/
  schema.sql               ← السكيمة الأساسية + قواعد الأمان (RLS) — تُشغّل مرة وحدة
  002_admin.sql             ← صلاحية الأدمن — تُشغّل بعد schema.sql
  seed.mjs                  ← سكربت لرفع كل القصص لقاعدة البيانات
  seed-accounts.mjs         ← ينشئ حساب أدمن + حساب ولي أمر تجريبي مع أطفال
  sync-image-urls.mjs       ← يربط صور Supabase Storage بقاعدة البيانات تلقائياً
src/
  lib/supabaseClient.js   ← اتصال Supabase من الواجهة
  data/stories.json        ← نفس بيانات القصص (مصدر التعبئة، ما عاد يُستخدم مباشرة بالواجهة)
  context/AppContext.jsx  ← الحالة العامة، الآن متصلة بـ Supabase (مصادقة + قاعدة بيانات حقيقية)
  components/             ← عناصر مشتركة
  pages/                  ← كل شاشة من شاشات رُواء (منها AdminDashboard.jsx)
```

### آلية عمل المصادقة الآن

- **تسجيل حساب** → `supabase.auth.signUp()` → Supabase يرسل بريد فيه رمز تحقق.
- **التحقق من البريد** → `supabase.auth.verifyOtp()` → عند النجاح يُنشئ Supabase جلسة (Session) تلقائياً.
- **إعداد ولي الأمر / إضافة طفل** → تُكتب مباشرة بجداول `profiles` و `children` (صف `profiles` يُنشأ تلقائياً عبر Trigger عند التسجيل).
- **المفضلة وتقدم القراءة** → مرتبطة بالطفل النشط (`activeChildId`) وتُقرأ/تُكتب من جداول `favorites` و `reading_progress` مباشرة.
- **تسجيل الخروج** → `supabase.auth.signOut()`.

كل هذا يحل محل التخزين المحلي (localStorage) اللي كان يستخدمه المشروع بالمرحلة السابقة.

### الصور التوضيحية للقصص

عندك طريقتين، والكود يدعمهم الاثنين بنفس الوقت (يعطي الأولوية لـ Supabase Storage لو موجودة، وإلا يرجع للملفات المحلية):

#### الطريقة 1: ملفات محلية بالمشروع (الأبسط)

```
public/images/stories/{story_id}/cover.(jpg|png|webp|jpeg)
public/images/stories/{story_id}/page-{page_number}.(jpg|png|webp|jpeg)
public/images/stories/{story_id}/back.(jpg|png|webp|jpeg)
```

**أي امتداد يشتغل** — الكود يجرب `jpg` ثم `png` ثم `webp` ثم `jpeg` تلقائياً. راجع `IMAGE_MAPPING.csv` لقائمة كل صورة متوقعة مع الـ `image_prompt` الأصلي.

#### الطريقة 2: رفع من لوحة Supabase مباشرة (بدون كود، بدون إعادة نشر)

1. **Storage** (بالقائمة الجانبية اليسرى) → **New bucket** → اسمه `story-images` → فعّلي **Public bucket** → Create.
2. جوا الـ bucket، أنشئي مجلد بنفس رقم كل قصة (`1`, `2`, `3`... مطابق لعمود `id` بجدول `stories`)، وارفعي داخل كل مجلد:
   - `cover.jpg` (أو png/webp)
   - `page-1.jpg`, `page-2.jpg`, ...
   - `back.jpg`
   (سحب وإفلات مباشرة من جهازك — نفس التسمية بالضبط المطلوبة بالطريقة 1)
3. بعد الرفع، شغّلي من جهازك:
   ```bash
   npm run sync:images
   ```
   يمسح السكربت كل الـ bucket ويكتب رابط كل صورة بعمود `cover_image_url` / `back_image_url` بجدول `stories` وعمود `image_url` بجدول `story_pages` تلقائياً.
4. حدّثي المتصفح — الصور تظهر مباشرة، بدون أي تعديل كود أو إعادة نشر.

كل مرة تضيفين أو تستبدلين صور بالـ bucket، شغّلي `npm run sync:images` من جديد.

## الشاشات المتصلة الآن بالباك-إند بالكامل

- إنشاء حساب → التحقق من البريد (رمز حقيقي) → إعداد ولي الأمر → إضافة طفل
- تسجيل الدخول / الخروج
- الصفحة الرئيسية، المكتبة، مشغل القصة — تُحمّل القصص من قاعدة البيانات
- المفضلة وتتبع تقدم القراءة — لكل طفل، محفوظة بقاعدة البيانات
- الملف الشخصي والإعدادات، لوحة تحكم ولي الأمر، تقارير الطفل — تعرض بيانات حقيقية للأطفال المسجّلين

**ملاحظة:** أرقام إحصائيات "تقارير الطفل" (متوسط الجلسة، المفردات المكتسبة، نشاط القراءة الأسبوعي...) لا تزال بيانات Mock تجميلية (`src/utils/mockStats.js`) لأنها تحتاج تتبع تفصيلي لكل صفحة/جلسة قراءة غير مُنفذ بعد — الأساسيات (المفضلة، آخر صفحة وصلها الطفل) حقيقية ومحفوظة فعلاً.

## الخطوات القادمة (غير مشمولة بهذه المرحلة)

- تفعيل "إنشاء قصة بالذكاء الاصطناعي" (يحتاج مفتاح API لنموذج LLM + خدمة مراجعة محتوى).
- تتبع تفصيلي لكل جلسة قراءة (وقت البداية/النهاية، الكلمات المقروءة) لتغذية تقارير حقيقية بدل الأرقام التجميلية.
- شاشات الاشتراكات والأوسمة/الإنجازات (موجودة بالتصاميم لكن لم تُبنَ بعد).

## تحديث الهوية والأدمن (المرحلة الأخيرة)

### الألوان — هويتين منفصلتين بنفس المشروع

- **`rawaa-*`** (بملف `tailwind.config.js`): الهوية الدافئة (أحمر/بنفسجي/ذهبي) لواجهة رُواء اللي يشوفها الأهل والأطفال — نفس عائلة الألوان القديمة تقريباً، بس منسّقة لتطابق المرجع البصري الجديد.
- **`medad-*`**: أزرق مداد الرسمي (`#0079CA` وما حولها) — **تُستخدم فقط بلوحة تحكم الأدمن**، حسب نظام هوية مداد (MEDAD Design System) المرفق. لا تظهر هذي الألوان لأي مستخدم عادي.
- الشعار بالهيدر وبشاشات الدخول تبدّل بالكامل لشعار مداد (`src/components/MedadLogo.jsx`) — شكل تقريبي (أشعة + كلمة "مداد") مبني بالكود نفسه، لأنه ما وصلني ملف شعار رسمي (SVG/PNG). لو عندك الملف الأصلي، أرفقه وأستبدله مباشرة بدل الرسم التقريبي.
- صورة البطل (الطفل + الجمل + النمر) محفوظة بـ `public/images/brand/hero-illustration.png` وتظهر بأسفل صندوق الهيرو بالصفحة الرئيسية (النص فوق، الصورة تحت، بنفس المربع) — مطابقة للمرجع اللي أرسلته.

### لوحة الأدمن — مُعاد بناؤها بالكامل

- شِل مستقل (`AdminShell.jsx`) بدون أي روابط رواء (لا مكتبة، لا ملفات أطفال، لا إنشاء قصة) — الأدمن يشوف لوحته فقط، ودخوله ما يفرض عليه تسجيل طفل.
- ٣ تبويبات:
  1. **نظرة عامة والحسابات** (كانت موجودة، محسّنة بألوان مداد).
  2. **مراجعة القصص المولدة** — خانة جاهزة بعلامة "قريباً" (تُفعَّل فعلياً بعد بناء ميزة توليد القصص بالذكاء الاصطناعي).
  3. **الشكاوى والملاحظات** — تعرض كل الرسائل اللي يرسلها أولياء الأمور، مع إمكانية تغيير حالة كل رسالة (جديد/قيد المعالجة/تم الحل).
- **"تسجيل دخول كهذا الحساب"** (Impersonation): زر بجدول الحسابات يفتح تبويب جديد يسجّل الأدمن دخوله تلقائياً بحساب ولي الأمر المحدد، عشان يقدر يشخّص أي مشكلة يبلغ عنها المستخدم. هذي الميزة **تحتاج خطوة نشر إضافية** (تحت).

### نموذج الشكاوى (لولي الأمر)

تبويب جديد "الدعم والملاحظات" داخل **الملف الشخصي والإعدادات** — يقدر ولي الأمر يرسل ملاحظة/شكوى/اقتراح، ويشوف حالة رسائله السابقة.

### خطوات إعداد إضافية مطلوبة الآن

1. شغّل **`supabase/003_feedback.sql`** بنفس طريقة الملفات السابقة (SQL Editor → لصق → Run) — يضيف جدول الشكاوى.

2. **لتفعيل ميزة "تسجيل الدخول كحساب آخر"**، تحتاج تنشر Edge Function وحدة (مرة وحدة فقط):

   ```bash
   npm install -g supabase        # لو ما عندك Supabase CLI مثبت
   supabase login
   supabase link --project-ref your-project-ref
   supabase functions deploy impersonate
   ```

   ما تحتاج تضبط أي متغيرات بيئة يدوياً — Supabase يوفر مفاتيح المشروع تلقائياً داخل الـ Edge Function. بعد النشر، الزر بلوحة الأدمن يشتغل مباشرة.

   > ⚠️ هذي الميزة **ما تقدر تشتغل بدون هالخطوة** — بدونها بيطلع خطأ واضح بالزر يوضح إنك تحتاج تنشر الـ function.

### الإنجازات — محسوبة فعلياً الآن (مو أرقام وهمية)

`src/utils/achievements.js` يحسب كل شارة من بيانات حقيقية (تقدم القراءة الفعلي للطفل النشط):

| الشارة | الشرط | ملاحظة |
|---|---|---|
| قارئ مبتدئ | أكمل 3 قصص | محسوب فعلياً |
| مستكشف | 5 قصص من 5 تصنيفات (`type`) مختلفة | محسوب فعلياً |
| قارئ نشيط | أكمل 10 قصص | محسوب فعلياً |
| مبدع / صانع الحكايات | 3 / 10 قصص بالذكاء الاصطناعي | **مقفلة دائماً** لحد ما تُبنى ميزة التوليد |
| نجم القراءة | 7 أيام قراءة متتالية | محسوب من تواريخ `reading_progress` |
| أسطورة القراءة | أكمل 50 قصة | محسوب فعلياً |
| جامع القيم | قصة واحدة على الأقل لكل قيمة من 5 (الصدق، الأمانة، التعاون، الرحمة، الشجاعة) | **تصنيف تقريبي بالكلمات المفتاحية** من نص `moral` — القصص اللي ما تحتوي كلمة مطابقة ما تُحتسب. يُفضّل مراجعتها يدوياً لاحقاً وربطها بعمود مخصص بقاعدة البيانات. |
| سفير رُواء | جمع كل الإنجازات | **مقفلة دائماً** (تعتمد على شارتين مقفولتين) |

### المكتبة — فلاتر جديدة

- **الفئة العمرية**: ٣ فئات نظيفة فقط (٤-٦ / ٧-٩ / ١٠-١٢ سنة) بدل النطاقات المتداخلة الأصلية. كل قصة تُصنَّف تلقائياً لأقرب فئة حسب أكبر تداخل بين نطاقها الأصلي والفئات الثلاث (`src/utils/ageBuckets.js`) — تقريبي، وتقدر تعدّله يدوياً لاحقاً لو حبيت.
- **نوع القصة**: فلتر إضافي حسب عمود `type` كما هو بالبيانات (11 نوع نصي مختلف حالياً).

### تسجيل الدخول لا يفرض إضافة طفل بعد الآن

فقط **التسجيل الأول** (إنشاء حساب جديد) يفرض على ولي الأمر إضافة أول طفل (بالمسار: تسجيل → تحقق → إعداد ولي الأمر → إضافة طفل). أي تسجيل دخول لاحق **ما يعيد فرضها** — لو صار عند ولي الأمر صفر أطفال لأي سبب، يقدر يتصفح الموقع عادي ويضيف طفل بعدين من "ملفات الأطفال" بالقائمة الرئيسية.

## تحديث: وقت الشاشة + تنبيهات تلقائية + تقارير حقيقية

### خطوة إعداد إضافية

شغّل **`supabase/004_screen_time.sql`** (نفس طريقة الملفات السابقة) — يضيف:
- `daily_screen_time_minutes` على جدول `children` (الحد اليومي المسموح، فارغ = بدون حد).
- `time_spent_seconds` على جدول `reading_progress` (وقت القراءة الفعلي المتراكم لكل قصة).

### وقت الشاشة (لوحة تحكم ولي الأمر → "الوقت المسموح بالموقع")

- ولي الأمر يحدد عدد الساعات/الدقائق المسموحة يومياً لكل طفل.
- الموقع يتتبع وقت الاستخدام الفعلي (يُحفظ محلياً بالمتصفح، يتجدد تلقائياً كل يوم).
- عند بلوغ الحد، يظهر **قفل كامل الشاشة** يطلب من الطفل تسليم الجهاز لولي الأمر — الخروج الوحيد هو تسجيل الخروج (يعيد الدخول ولي الأمر لاحقاً أو يزيد الحد لغداً).
- بدون حد محدد (فارغ) = استخدام غير محدود، بدون أي قفل.

### التنبيهات — صارت تلقائية بالكامل

كل تنبيه يُبنى من بيانات حقيقية (`src/utils/notifications.js`):
- **وسام جديد**: يظهر لكل شارة إنجاز حقّقها الطفل فعلياً (من محرك الإنجازات الحقيقي).
- **محتوى جديد**: يظهر تلقائياً لو انضافت قصص للمكتبة خلال آخر 7 أيام.

> ملاحظة: التنبيهات مبنية على الطفل **النشط حالياً** (المحدد بأعلى الصفحة) فقط، لأن باقي البيانات (تقدم القراءة) تُجلب لطفل واحد بكل مرة لتقليل عدد الاستعلامات — لو عندك أكثر من طفل، بدّل الطفل النشط لتشوف تنبيهاته.

### تقرير الطفل — بيانات حقيقية بدل الأرقام التجميلية

`src/utils/childReportStats.js` يحسب من بيانات القراءة الفعلية:
- **النوع المفضل**: أكثر تصنيف قصص (`type`) قرأه الطفل فعلياً.
- **مفردات مكتسبة**: عدد الكلمات العربية الفريدة بكل القصص اللي بدأ الطفل قراءتها.
- **متوسط الجلسة**: من وقت القراءة الفعلي المتتبَّع (`time_spent_seconds`) — يظهر "لا توجد بيانات بعد" لو الطفل ما قرأ شي بعد تفعيل هذا التتبع.
- **نشاط القراءة الأسبوعي**: عدد مرات تحديث تقدم القراءة الفعلية لكل يوم بالأسبوع (إشارة حقيقية، مو دقائق تقديرية).

**ملاحظة صادقة:** "مستوى الصداقة" والوصف السردي بجانب الشخصية المفضلة لسا Mock — ما فيه إشارة بيانات حقيقية توضح "الشخصية المفضلة" فعلياً (يحتاج مفهوم/تتبع جديد لهذا لاحقاً).

### إزالات

- حُذفت "إحصائيات القراءة الأسبوعية" من **لوحة تحكم ولي الأمر** (موجودة بشكل أدق وأصح بصفحة تقرير الطفل).
- حُذفت بطاقة "إضافة طفل آخر" من **تقرير الطفل** (موجودة أصلاً بلوحة تحكم ولي الأمر).

## تحديث: السرد الصوتي (TTS) — صوت واحد حالياً

### كيف تشتغل الميزة

- زر التشغيل بمشغّل القصة يولّد صوت الصفحة الحالية **أول مرة بس** (عبر ElevenLabs)، ويخزّنه دائماً بـ Supabase Storage.
- أي طفل ثاني (أو نفس الطفل) يفتح نفس الصفحة بعدها، يسمع الصوت المخزّن فوراً — بدون أي تكلفة إضافية أو انتظار.
- سرعة القراءة (٠.٥x – ٢x) تتحكم فيها فعلياً من لوحة الإعدادات.
- الأزرار ⟲15 / 15⟳ ترجع/تقدّم بالصوت الفعلي.
- شلت مؤقتاً قوائم اختيار الراوي/اللهجة (كانت شكلية بس) لحد ما نضيف أصوات حقيقية إضافية لاحقاً — صوت واحد الآن.

### خطوات الإعداد (مرة وحدة)

**1) شغّلي `supabase/006_narration.sql`** (نفس طريقة الملفات السابقة).

**2) أنشئي Bucket جديد بـ Supabase Storage:**
Storage → New bucket → اسمه بالضبط `story-audio` → فعّلي **Public bucket** → Create.

**3) خذي مفتاح API من ElevenLabs:**
[elevenlabs.io](https://elevenlabs.io) → حساب → **Profile → API Keys** → انسخي المفتاح.

**4) اختاري صوتاً (Voice ID):**
من لوحة ElevenLabs → **Voice Library** → اختاري صوت عربي مناسب → انسخي الـ `Voice ID` تبعه (يظهر بتفاصيل الصوت).

**5) سجّلي المفتاحين كـ Secrets بمشروع Supabase:**
```bash
supabase secrets set ELEVENLABS_API_KEY=your_key_here
supabase secrets set ELEVENLABS_VOICE_ID=your_voice_id_here
```

**6) انشري الـ Edge Function:**
```bash
supabase functions deploy tts
```

بعد هذي الخطوات، زر التشغيل بمشغّل القصة يشتغل فعلياً.

### لو تبين تبدّلين لـ Gemini TTS لاحقاً

كل كود الربط بـ ElevenLabs معزول بمكان واحد بس: دالة `synthesizeSpeech()` بأول ملف `supabase/functions/tts/index.ts`. تبدّلين محتوى هذي الدالة بس (تتصلين بـ Gemini API بدل ElevenLabs) — الباقي كله (التخزين، الـ Cache، الواجهة، قاعدة البيانات) يبقى زي ما هو بدون أي تعديل.
